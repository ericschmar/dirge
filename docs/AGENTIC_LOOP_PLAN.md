# Agentic Loop — Feature Audit & Phased Plan

Comparing `agentic-features.md` against the current code state on
`main`. Many of the catalogued features already exist (often
landed via prior audit rounds under different names). The plan
below focuses on the genuine gaps and prioritises by leverage on
real failure modes.

## Phased Plan

### Phase 1 — Plumbing & telemetry (~1 day, low-risk)

**Goal**: tighten what exists, get measurable signal before adding new features.

1. **Aggregate repair-rate counters** — add an `AtomicU64` per
   `RepairKind` updated by `validate_and_repair` and dumped at
   `AgentEnd` via a `LoopEvent::RepairStats { kind, count }`. The
   UI logs them when the session closes; future dashboard is just
   reading the tracing field.
2. **`tool_input_invalid` structured log** — when repair exhausts,
   emit a dedicated `tracing::warn!` with `tool`, `model`,
   `original_args` (untruncated up to 16 KiB), and the validator
   error list. Currently lives inside a generic `tool_repair =
   "failed"` log.
3. **Tool description-hint helper** — small `fn
   contract_hint_for(tool)` that prepends a one-liner contract
   note onto every built-in tool description ("path is an
   absolute filesystem path, not a markdown link"). Already
   true for `read`/`write`; centralise so adding a tool is
   one line.
4. **Streaming-chunk timeout** — wire a `chunk_timeout` parameter
   to `tool_input_repair`'s reassembly so a stuck partial-JSON
   payload surfaces a clean error instead of waiting on the
   provider's outer 300s budget.

**Deliverable**: every existing repair surface produces structured
telemetry; tool descriptions enforce contract hints uniformly.

### Phase 2 — Schema annotations + tree-sitter output validation (~2-3 days)

**Goal**: replace per-tool heuristics with schema-declared
semantics; catch generated-code syntax errors at the write
boundary, not after the LLM has built on top of broken code.

1. **`SchemaHint` extension** — extend the JSON schemas that
   `Tool::definition` returns with a `dirge-hints` object:
   ```json
   {"path": {"type": "string", "dirge-hints": {"semantic": "absolute_path"}}}
   ```
   Variants: `absolute_path`, `relative_path`, `non_empty_array`,
   `relational_requires: ["offset", "limit"]`. The repair layer
   reads `dirge-hints` first, falls back to field-name heuristic
   for backward compat.
2. **Generic relational defaults** — when a hint declares
   `relational_requires`, the harness defaults the missing
   counterpart AND prepends a `Note:` to the tool result via a
   small post-execute hook. Removes the `read.rs:190` hardcoding.
3. **Tree-sitter validation on `write` / `edit` / `apply_patch`** —
   before the atomic write, parse the new file content with the
   appropriate tree-sitter adapter (`src/semantic/adapters/`).
   If parsing surfaces ERROR nodes inside the dirty region,
   return the tool result with `is_error: true` + the syntactic
   error list. Default-on for known-supported languages
   (`.rs`/`.ts`/`.py`/`.go`/`.cpp`/etc.). Off-by-default for
   unrecognised file types.

**Deliverable**: a new tool author annotates the schema and gets
auto-repair + relational defaults for free; the agent can't
write syntactically-broken code without seeing the error in the
same turn.

### Phase 3 — Context management (~2-3 days) ✅ shipped

**Goal**: stop sending the entire toolset every turn; stop
dumping multi-MB tool outputs into the context.

1. **`tool_search` meta-tool** — built-in tool that takes a
   `query: string` and returns the top-K most-similar tool
   names + descriptions. Implementation: keep the full
   tool-defs registry but mark them as "available on request"
   in the system prompt; only `tool_search` itself is unconditionally
   present. After the model calls `tool_search`, the harness
   injects the selected tools' full schemas into the next turn.
   Major token savings on long sessions; needs careful UX so
   the model doesn't get confused.
2. **Disk-backed large-output relay** — replace the current
   bash/webfetch truncation with: write the full output to
   `~/.dirge/transient/<turn_id>/<tool>.txt`, return a summary
   (first N lines + last M lines + line count + file path).
   Add a hint suggesting `read` with offset/limit for details.
   The model agency Hermes/pi both use.
3. **Anthropic prompt-cache positioning audit** — verify that
   the system prompt + tool defs land at the start of the
   payload (cache-warm) and dynamic content (history, current
   prompt) lands at the end. Add a small assertion in the
   request builder so a future refactor can't accidentally
   reshuffle.

**Deliverable**: 100-turn session uses ~30% fewer tokens; large
outputs no longer overflow context.

### Phase 4 — Cognitive routing + context depth (~3-4 days)

**Goal**: keep the agent grounded over long runs.

1. **Dual-client tiering** — extend `AnyClient` so a single
   session can route different turns through different models.
   Default to the user's configured model; on repair-exhaustion
   OR tree-sitter syntactic-failure in the same tool call,
   escalate the NEXT retry to a more capable tier (configurable
   per provider, e.g., `deepseek-v4-pro` if main is `flash`).
   Surface the escalation as a `Note:` in the tool result so
   the user can see it happened.
2. **Context-depth reminder system** — count consecutive turns
   that touched the same file (`session_search` already gives
   us the data). When the count crosses a threshold (e.g. 8),
   inject a system-reminder summarising the active task + key
   decisions. Counter resets when the user submits a fresh
   prompt that doesn't mention the same files.
3. **Programmatic tool calling** (stretch) — let the model emit
   a Janet (or restricted Python) snippet that orchestrates
   multiple tool calls. Plugin system already runs Janet on a
   worker thread; binding tool-call APIs into the Janet context
   is the bulk of the work. Could land standalone after
   Phase 4.

**Deliverable**: 1000-turn session stays on task; spec'd budget
escalation handles deep-semantic failures without manual
intervention.

## Risk + sequencing notes

- **Phase 1 is the lowest-risk, highest-leverage step** because
  it makes everything else measurable. Don't skip it just
  because the features sound boring.
- **Phase 2's tree-sitter validation has a sharp edge**:
  default-on for a language we don't have an adapter for would
  break writes. Gate it strictly on `registry.find_for_file_with_content`
  returning `Some`.
- **Phase 3's `tool_search`** can DEGRADE performance on
  short-session benchmarks because the model needs an extra
  round-trip. Default-off behind a config flag; enable it for
  sessions with > 20 tool defs (MCP-heavy users).
- **Phase 4's cognitive routing is the biggest UX risk** — if
  the escalation happens behind the user's back they'll notice
  unexpected token spend. Always surface the model swap.

## Out of scope

- The published-as-library "Phase 4" from agentic-features.md.
  dirge is a binary; the repair layer is reusable in principle
  but extracting it as a separate crate is a bigger project.
- Per-model dashboards beyond what tracing logs give us. The
  ecosystem (Loki, Honeycomb, etc.) can ingest the existing
  events.
